데이터 사이언티스트, 엔지니어, 분석가 모두 데이터 기반으로 의사결정을 지원한다는 공통점이 있지만, 각자의 역할과 집중하는 영역에는 분명히 차이가 있다고 생각한다. 
따라서 SQL의 사용도 활용 목적에 따라 다를 것이라고 생각이 드는데, 분석을 위한 SQL은 SELECT 문 (흔히 집계, 필터링, 조인)을 활용해서 의미 있는 지표만 빠르게 추출하는 것이 주로 초점이라면,
개발이나 엔지니어링에서의 SQL은 데이터의 구조 설계, 디비 스키마 설계, 성능 최적화와 같은 시스템 적인 측면이 더 강조되기에 이 중심의 접근이 다르다고 생각한다. 
따라서 데이터 사이언티스트 입장에서 단순히 모델을 만들거나 분석만 하는 역할이 아니라, 추가적으로 데이터를 어떻게 수집하고 가공하면서 저장하는지에 대한 전체적인 데이터 파이프라인을 이해하는 것이 중요하다고 느꼈다.
특히 데이터 엔지니어링적인 SQL 활용 능력도 같이 추가되며 데이터 파이프라인 구조와 BI도구를 활용하는 방법까지 기르며 역량을 키워나갈 것이다.

## 02. 시스템

<!-- 교재에서 나열된 도구들의 특징을 정리해주세요. -->

#### 1. PostgreSQL

- **오픈소스 RDB (Relational Database)** 
- GUI 인스톨러의 제공으로 로컬에서 쉽게 설치하면서 사용 가능
- MySQL과 비교할 때 표준 SQL을 잘 준수하며 미들웨어 중에서 역사가 가장 오래되었으며, 확장 기능이 많다. 



#### 2. Apache Hive

**HDFS(Hadoop File System)** 이라고 부르는 파일 시스템 위의 데이터를 SQL 스러운 인터페이스로 간단하게 처리해주는 시스템 

- 디스크에서 데이터를 읽어 들일 때의 전송 속도 -> CPU에서 처리할 수 있는 속도에 비해 매우 느리다. 
  - 이를 **고속으로 처리하기 위한 아키텍처로 분산 파일 시스템이 고안**
  - 분산 파일 시스템 위의 데이터를 순서를 맞추는 방법으로 **MapReduce**라는 알고리즘이 고안
- **Hive 특징** : 어디까지나 파일 기반의 시스템에 있다는 것을 들 수 있다. 
  - 파일 기반 시스템이므로 특정 레코드 하나를 변경하거나 제거하는 것이 어렵고, 인덱스도 디폴트로 존재하지 않아 쿼리 실행 때 파일 전체를 조작해야 함. 
  - 쿼리 실행 때 동적으로 데이터를 정의할 수 있음. 
  - UDF(User-Defined Function) 를 활용해서 SQL만으로는 구현하기 어려운 문자열 처리 간단하게 가능 
- **스루풋을 높이기 위한 아키텍처를 가지고 있어 리액턴시가 낮은 처리를 요구하는 경우**에는 적합 X
  - Spark, Tez으로 사용하거나 데이터 형식을 최적화해서 리액턴시를 낮추는 방법도 쓰임. 



#### 3. Amazon Redshift

**AWS에서 제공하는 분산 병렬 RDB로 아예 RDB**

- 레코드를 업데이트하거나 제거할 수도 있고, 트랜잭션도 지원 
- RDB에서 다룰 수 없는 대량의 데이터와 상호 작용하는 쿼리를 실행하고 싶을 때 효과적
  - **클라우드 서비스**이므로 초기에는 작은 규모로 시작하고  데이터가 많아지면 스케일아웃해서 큰 규모로 쉽게 변경할 수 있다는 장점이 존재
- 성능 튜닝을 하거나 비용을 줄이기 위해서 최적의 노드 수와 스펙을 예측해서 인스턴스의 실행과 종료를 관리해야 함. (단점)
- **컬럼 지향 스토리지** : 레코드별로 저장하는 것이 아니라 컬럼별로 저장하는 아키텍처
  - 데이터의 압축률을 높이고, 쿼리 실행 시 디스크 I/O를 줄이기가 가능



#### 4. Google BigQuery

**빅데이터 분석을 위해 구글이 제공하는 클라우드 서비스**

- 직접 노드 인스턴스를 관리할 필요가 없고 사용 시간이 아닌 읽어 들인 데이터의 양으로 비용이 발생함.
  - 따라서 사용 요금을 생각보다 예측하기 어렵다는 단점이 존재
- **RedShift와 다른 점** : 직접 노드 인스턴스를 관리할 필요가 없다. + 구글이 제공하는 다른 클라우드 서비스와도 쉽게 연동이 가능하다. 
- BigQuery는 `레거시 SQL` 과 `스탠다드 SQL (표준 SQL)` 의 2종류로 나뉜다. 
  - 쿼리 실행 시의 옵션과 어노테이션을 사용해 간단하게 스탠다드 SQL로 변환해서 사용이 가능 



#### 5. SparkSQL

**Apache Spark의 기능 중에서 SQL 인터페이스와 관련된 기능을 나타내는 용어**

- 무료로 사용이 가능하고, 오픈소스 제품 중에서 가장 빠른 속도로 개발이 이루어지는 제품
- 장점 : **빅데이터 활용과 관련된 대부분의 처리를 한 번에 구현할 수 있다는 점**
  - 인터페이스로 프로그래밍 언어를 지원하기에 export, import 기능이 다양함. 
- **Spark 2.0** : API가 표준으로 추가 -> 비슷한 형태로 프로그래밍 작성이 가능하고 성능 문제 등도 자동으로 최적화 
  - 데이터 처리를 작은 모듈로 분할해서 다양한 처리 가능

### 데이터의 종류

#### 1. 업무 데이터 

**서비스와 시스템을 운용하기 위한 목적으로 구축된 데이터베이스에 존재하는 데이터**

- 대부분이 **갱신형 데이터**
- 두 가지 데이터로 분류 가능
  - **트랜잭션 데이터** : 서비스와 시스템을 통해 사용자의 행동을 기록한 데이터로 리포트 제작할 때는 설명하는 마스터 데이터가 필요함.
  - **마스터 데이터** : 서비스와 시스템이 정의하고 있는 데이터 

**업무 데이터의 특징**

**1. 데이터의 정밀도가 높다.**

- 트랜잭션과 롤백의 기능이 있어 업무 처리의 문제 해결이 가능하고 데이터의 정합성이 보증
- 정확한 값이 요구되는 매출 관련 리포트에서 사용

**2. 갱신형 데이터** 

**3. 다뤄야 하는 테이블의 수가 많다.**

- 대부분의 서비스는 RDB사용 
  - 확장성 배제, 데이터의 정합성을 쉽게 유지하여 데이터를 저장하기 위함. 

**4. 업무 데이터 축적 방법**

- RDB에서 빅데이터 분석 기반으로 데이터 로드를 해주는 시스템을 사용해야 함. 

**5. 모든 데이터 변경하기**

- 날짜를 기반으로 누적되는 경우가 아니라면, 전체를 한꺼번에 바꾸어 최신 상태로 만들어야 함. 
  - 변경 시에는 과거의 정보를 잃지 않기 위해 주의가 필요

**6. 모든 레코드의 스냅샷을 날짜별로 저장하기**

**7. 어제와의 변경 사항만 누적하기**



**업무 데이터 다루는 방법**

- 트랜잭션 기능으로 데이터의 정합성이 보장되기에 **정확한 값을 요구할 경우에 활용해야 함.**
  - 다만 서비스의 방문 횟수, 페이지뷰, 사용자 유도 등의 업무 데이터가 아닌 로그 데이터와의 구분이 필요
- 데이터 변경이 발생할 수 있기에 **추출 시점에 따라 결과의 변화의 여부 파악**
  - 데이터 변경의 영향을 최소화하면서 축적하는 방법이 좋다. 
  - 기본적으로 신뢰가 가는 데이터 -> **정합성 보장이 Key 포인트**



#### 2. 로그 데이터

**통계 또는 분석을 주 용도로 설계된, 특정 태그를 포함해서 전송된, 특정 행동을 서버 측에 출력한 데이터**

- **누적형** 데이터로 출력 시점의 정보를 축적해두는 것 
- 로그 출력 이후 가격이 변경되거나 사용자 정보가 변경되더라도 기존의 데이터를 수정하지 않는다. 



**로그 데이터의 특징**

**1. 시간, 사용자 엔드 포인트, IP, URL, 레퍼러, Cookie 등의 정보 저장**

**2. 추출 방법에 따라 데이터의 정밀도가 달라짐**

**3. 계속 기록 추가 -> 과거 데이터에 영향 X**



**로그 데이터의 축적 방법**

- 태그, SDK 를 통해 **사용자 장치에서 데이터를 전송하고 출력하기**
  - 웹 사이트 개발 시에 활용하는 굉장히 일반적인 방법

- 서버에서 **데이터를 추출하고 출력하기**
  - 별도의 처리 없이 서버에서 로그를 출력하는 방법
  - 클롤러의 접근을 막는 것은 거의 불가능 (매일매일 새로운 크롤러가 나오기 때문)



**로그 데이터 다루는 방법**

- 사이트 방문 횟수, 페이지 뷰, 사용자 유도 상황을 집계하고 분석할 때 주로 사용
- 최신 상태를 고려한 분석에는 적합하지가 않다. 
  - 기록을 누적하는 형태이기 때문에 **추출 결과의 변할 가능성이 적다.**
- **데이터의 정확도는 업무 데이터에 비해 낮다.**
  - 누락 가능성, 크롤러의 로그가 집계 등의 위험이 존재 



### 두 데이터를 사용해서 생성되는 가치

- 목표를 관리, 설계, 서비스/조직의 성장에 기여하기 
- 사용자 행동을 기반으로 경향을 발견하고, 매출과 서비스 개선에 기여하기 
- 과거의 경향을 ㅣ반으로 미래의 행동 예측하기 
  - 추천 시스템 같은 부분
